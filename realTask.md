# 产品专家 AI 平台 OpenClaw 工作流真实需求

> 来源：`文档 (33).docx`
>
> 目标：把原始“产品专家 AI 平台 V8.0”方案收敛为可落地的 OpenClaw 工作流需求。最终交付不是先做完整平台 UI，而是先交付一组可运行的 `skill`、`openclaw cron` 定时任务和 `memory.md` 龙虾记忆。

## 1. 需求总述

要建设一个基于飞书、GDP 数据底座和龙虾 OpenClaw 的产品专家 AI 工作流系统，用于支持销售增长、方案生成、客户赋能、储能 ROI 测算、交付运维和经营复盘。

系统需要把客户从“模糊需求”推进到“可生成方案、可测算收益、可客户承认、可转订单、可交付、可复盘”的标准流程。

核心业务闭环：

```text
客户信息采集
-> 项目需求明确
-> 应用场景明确
-> 年度需求明确
-> 采购计划明确
-> 储能 ROI 测算
-> 生成《客户需求与项目说明书》
-> 生成标准 Word 技术方案书
-> 方案工程审核
-> 客户赋能
-> 客户承认
-> 报价、合同、订单转化
-> 交付工程实施
-> FAE 运维服务
-> 实际运行数据回流
-> ROI 模型校准
-> 管理者复盘
-> 反向优化产品、标准包、模板、销售打法和投资模型
```

## 2. 最终交付范围

本阶段交付应收敛为三类 OpenClaw 资产。

### 2.1 Skill 交付

需要交付一组 OpenClaw skills，每个 skill 负责一个稳定业务动作。

建议 skill 清单：

| Skill | 作用 |
| --- | --- |
| `demand-intake-skill` | 采集和结构化客户、项目、场景、年度需求、采购计划 |
| `demand-gate-skill` | 计算需求完整度评分，识别缺失项，生成澄清问题 |
| `customer-profile-skill` | 从 GDP 读取客户画像、历史方案、历史订单、行业和区域信息 |
| `scenario-detect-skill` | 判断新建、替换、扩容、改造、集成、试点、投标等应用场景 |
| `storage-project-detect-skill` | 判断是否为储能项目，并区分户用储能和工商储 |
| `home-storage-roi-skill` | 执行户用储能 ROI 初筛、标准测算和投资决策测算 |
| `ci-storage-roi-skill` | 执行工商储 ROI 初筛、标准测算和投资决策测算 |
| `requirement-brief-skill` | 生成《客户需求与项目说明书》 |
| `solution-word-draft-skill` | 按标准 Word 模板生成项目技术方案书初稿 |
| `standard-package-match-skill` | 根据应用场景匹配 15 类标准包 |
| `solution-risk-review-skill` | 检查技术、认证、交付、商务、ROI 和竞品风险 |
| `customer-enablement-skill` | 生成客户资料包、一页纸摘要、管理层材料、技术评审材料、FAQ、ROI 一页纸 |
| `delivery-handoff-skill` | 从已承认方案生成交付 9 件套、实施计划、验收标准、项目移交资料 |
| `fae-support-skill` | 支持技术问答、运维问答、故障排查、专家转接 |
| `faq-memory-skill` | 自动采集、分类、归并、升级 FAQ，并写入记忆 |
| `roi-calibration-skill` | 用实际运行数据校准 ROI 模型 |
| `review-report-skill` | 生成周报、月报、季报、年报和优化建议 |

### 2.2 OpenClaw Cron 交付

需要交付定时任务，用于自动巡检、沉淀、复盘和校准。

建议 cron 清单：

| Cron | 频率 | 作用 |
| --- | --- | --- |
| `sync-gdp-feishu-data` | 每天 | 同步客户、商机、订单、方案、服务、ROI、运行数据 |
| `scan-incomplete-demands` | 每天 | 扫描未通过准入的需求，生成缺失项提醒 |
| `daily-faq-collect` | 每天 | 汇总销售、客户、FAE 问答，沉淀 FAQ 候选 |
| `weekly-demand-quality-report` | 每周 | 输出需求完整率、退回原因、缺失字段分布 |
| `weekly-faq-report` | 每周 | 输出高频 FAQ、未解决问题、专家修正清单 |
| `weekly-sales-conversion-report` | 每周 | 复盘需求、方案、赋能、承认、订单转化 |
| `monthly-roi-report` | 每月 | 分析户用储能和工商储 ROI 表现 |
| `monthly-template-standard-package-report` | 每月 | 分析 Word 模板和 15 类标准包使用效果 |
| `quarterly-model-calibration` | 每季度 | 基于实际运行数据校准 ROI 模型 |
| `quarterly-business-review` | 每季度 | 复盘客户画像、应用场景、产品线、区域、销售打法 |
| `annual-knowledge-consolidation` | 每年 | 生成年度 FAQ、年度方案白皮书、ROI 报告、模板优化建议 |

### 2.3 `memory.md` 交付

需要提供一个长期维护的龙虾记忆文件，作为所有 skill 的业务上下文。

`memory.md` 至少包含：

| 模块 | 内容 |
| --- | --- |
| 平台定位 | 产品专家 AI 是销售增长、方案转化、客户赋能、储能投资决策和经营复盘平台 |
| 核心原则 | 先需求准入，再生成方案；先标准模板，再 AI 填充；先方案工程提交，再客户赋能；储能项目必须 ROI 测算 |
| 准入规则 | 客户信息、项目需求、应用场景、年度需求、采购计划、ROI 前置信息的必填规则 |
| 需求评分规则 | 完整度权重、评分阈值、系统动作 |
| ROI 风控规则 | 数据不足不输出正式结论；不确定收益单列；外发前必须审核 |
| 方案模板规则 | Word 主模板、模块化章节库、AI 只能填充，不能自由改目录 |
| 标准包规则 | 15 类标准包匹配逻辑、适用场景、引用方式 |
| 权限矩阵 | 销售、销售主管、方案工程师、FAE、交付工程师、产品经理、ROI 测算负责人、管理层、知识管理员、系统管理员权限 |
| 数据字典 | 客户需求主表、方案主表、ROI 测算主表字段定义 |
| FAQ 记忆 | 高频问题、专家修正、已确认答案、暂无依据的问题 |
| 复盘结论 | 周、月、季、年复盘沉淀出的模板、产品、ROI 模型、销售打法优化建议 |

## 4. 核心业务规则

### 4.1 方案生成准入规则

正式方案生成前必须完成：

- 客户信息
- 项目需求
- 应用场景
- 年度需求
- 采购计划
- 储能项目的 ROI 测算必要数据
- 《客户需求与项目说明书》

强约束：

- 没有完整客户需求，不生成正式方案。
- 没有应用场景，不匹配标准包。
- 没有年度需求，不生成年度解决方案。
- 没有采购计划，不进入转化预测。
- 没有 ROI 准入数据，不输出正式投资回报结论。
- 没有《客户需求与项目说明书》，不允许生成正式项目技术方案书。

### 4.2 需求完整度评分

| 维度 | 权重 |
| --- | --- |
| 客户信息完整度 | 15% |
| 项目需求清晰度 | 20% |
| 应用场景清晰度 | 20% |
| 年度需求清晰度 | 15% |
| 采购计划清晰度 | 15% |
| 风险与限制条件清晰度 | 10% |
| 储能 ROI 数据完整度 | 储能项目附加评分 |

评分动作：

| 分数 | 系统动作 |
| --- | --- |
| 90 分以上 | 可生成正式项目技术方案书 |
| 75-89 分 | 可生成方案初稿，但必须标记待确认项 |
| 60-74 分 | 不生成正式方案，只输出需求澄清清单和初步建议 |
| 60 分以下 | 不进入方案工程，要求销售补充基础信息 |

### 4.3 AI 使用边界

AI 只做初稿、建议、检查和计算，不替代专业审核。

必须人工审核的内容：

| 内容 | 审核角色 |
| --- | --- |
| 客户外发方案 | 方案工程师 |
| 技术参数承诺 | 方案工程师 / 产品专家 |
| 认证承诺 | 产品经理 / 认证负责人 |
| 交付周期 | 交付工程负责人 |
| 储能 ROI 正式结果 | 方案工程师 / 财务或投资测算负责人 |
| 竞品对比 | 产品经理 / 法务或合规负责人 |
| 合同级承诺 | 商务 / 法务 / 管理审批 |

## 5. 功能需求明细

### 5.1 客户需求采集与结构化

使用场景：

- 销售拿到客户线索后，需要把口头、微信、会议纪要中的模糊需求变成标准需求记录。
- 客户只给出大概想法，需要 AI 帮销售追问关键字段。
- 需求准备进入方案工程前，需要形成结构化输入。

输入数据：

- 客户名称、行业、区域、规模、等级、联系人、决策链
- 项目背景、痛点、目标、范围、功能、技术、认证、周期
- 应用场景：新建、替换、扩容、改造、集成、试点、投标
- 年度采购规模、分阶段计划、覆盖区域、产品线需求
- 采购时间、采购方式、预算状态、报价节点、合同节点

数据流程：

```text
销售输入原始客户信息
-> demand-intake-skill 抽取字段
-> customer-profile-skill 补充 GDP 客户画像
-> scenario-detect-skill 判断应用场景
-> demand-gate-skill 检查完整度
-> 输出客户需求记录、缺失项、访谈提纲
-> 写入客户需求主表和 memory.md 的客户/需求片段
```

具体要干的事：

- 把销售输入转成标准 JSON 或表单字段。
- 自动识别缺失字段。
- 自动生成客户访谈提纲。
- 自动生成客户画像摘要。
- 自动生成需求完整度评分。
- 合格后生成方案申请单。

输出物：

- 客户需求记录
- 客户画像摘要
- 需求完整度评分
- 缺失信息清单
- 客户访谈提纲
- 方案申请单

OpenClaw 交付：

- `demand-intake-skill`
- `customer-profile-skill`
- `scenario-detect-skill`
- `demand-gate-skill`
- `scan-incomplete-demands` cron
- `memory.md` 记录客户画像、需求规则、常见缺失项

### 5.2 需求准入与方案生成门禁

使用场景：

- 销售提交方案申请时，系统判断是否允许进入方案工程。
- 信息不完整时，不能直接让 AI 生成正式方案。
- 方案工程师需要快速知道待确认项。

输入数据：

- 客户需求主表字段
- 风险与限制条件
- 储能项目 ROI 前置数据
- 历史类似需求的缺失项和澄清问题

数据流程：

```text
客户需求记录
-> demand-gate-skill 按评分规则打分
-> 判断准入等级
-> 分支处理：
   90+：允许正式方案
   75-89：允许初稿，标记待确认项
   60-74：只输出澄清清单和初步建议
   <60：退回销售补充基础信息
-> 状态写回需求主表
-> 缺失项写入 memory.md 作为后续提醒样本
```

具体要干的事：

- 按权重计算需求完整度。
- 判断是否存在硬性缺失项。
- 生成“为什么不能生成正式方案”的解释。
- 给销售生成补充问题。
- 给方案工程师生成风险提示。

输出物：

- 准入结论
- 完整度评分
- 缺失字段列表
- 需求澄清清单
- 初步建议
- 方案生成允许状态

OpenClaw 交付：

- `demand-gate-skill`
- `scan-incomplete-demands` cron
- `weekly-demand-quality-report` cron
- `memory.md` 记录评分规则、硬性门禁和退回原因

### 5.3 储能项目识别与 ROI 测算准入

使用场景：

- 销售或方案工程师提交需求时，系统判断是否为储能项目。
- 储能项目需要先区分户用储能和工商储。
- ROI 数据不足时，不允许输出正式投资回报结论。

输入数据：

- 项目描述、产品线、应用场景
- 地区、电价、负荷、储能功率、储能容量、投资成本
- 光伏、补贴、需求响应、虚拟电厂等增值收益信息

数据流程：

```text
需求记录
-> storage-project-detect-skill 判断是否储能
-> 判断储能类型：户用储能 / 工商储
-> ROI 准入字段检查
-> 数据完整则调用对应 ROI skill
-> 数据不完整则输出缺失项和可用默认参数范围
-> ROI 结果写入 ROI 测算主表
-> 关键假设和风险写入 memory.md
```

具体要干的事：

- 识别储能项目类型。
- 校验户用储能和工商储的必填字段。
- 判断可输出的 ROI 等级：初步测算、标准测算、投资决策测算。
- 明确哪些收益可以计入保守收益，哪些只能作为增值收益。
- 标记不可外发或需审核的 ROI 结果。

输出物：

- 是否储能项目
- 储能项目类型
- ROI 数据完整度
- ROI 测算等级
- 缺失参数清单
- ROI 测算任务

OpenClaw 交付：

- `storage-project-detect-skill`
- `home-storage-roi-skill`
- `ci-storage-roi-skill`
- `memory.md` 记录 ROI 准入规则和风险口径

### 5.4 户用储能 ROI 测算

使用场景：

- 家庭用户、别墅用户、海外户储客户、分布式光伏业主、渠道年度客户需要判断储能投资是否值得。
- 销售早期需要初步回本判断。
- 方案工程阶段需要标准 ROI 章节。
- 客户内部评审需要 IRR、NPV、现金流和敏感性分析。

输入数据：

- 地区、户型、用电场景、年度需求、采购计划
- 月用电量、年用电量、日均用电量、峰谷比例
- 光伏容量、年发电量、无储能自用率、加储后自用率
- 储能容量、储能功率、DoD、系统效率、循环次数、衰减率
- 电池、逆变器、BMS/EMS、安装、并网、运维、保险成本
- 居民购电价、峰谷电价、上网电价、补贴、电价上涨率

数据流程：

```text
户用储能 ROI 输入
-> home-storage-roi-skill 校验必填项
-> 计算光伏自用提升收益
-> 计算峰谷套利收益
-> 计算余电上网、补贴、备电和可选虚拟电厂收益
-> 计算年度总收益和年度净收益
-> 计算静态回收期、动态回收期、ROI、IRR、NPV
-> 生成敏感性分析和投资建议
-> 写入 ROI 测算主表
-> 可写入方案书 ROI 章节和客户赋能材料
```

具体要干的事：

- 计算光伏自用提升收益：光伏自用提升电量 × 居民购电价与余电上网电价差。
- 计算峰谷套利收益：年放电量 × 峰谷价差 × 系统效率。
- 区分保守收益、基础收益、增值收益。
- 输出推荐、谨慎、暂不建议三类投资建议。
- 标注所有假设和风险边界。

输出物：

- 项目总投资
- 年度总收益
- 年度净收益
- 静态回收期
- 动态回收期
- ROI、IRR、NPV
- 光伏自用率提升
- 电费节省比例
- 敏感性分析
- 投资建议

OpenClaw 交付：

- `home-storage-roi-skill`
- `monthly-roi-report` cron
- `quarterly-model-calibration` cron
- `memory.md` 记录户储公式、默认假设、风险口径

### 5.5 工商储 ROI 测算

使用场景：

- 工厂、园区、商业综合体、充电站、数据中心、光伏工商业客户评估储能投资。
- 方案工程阶段需要正式收益模型。
- 客户内部评审需要现金流、IRR、NPV 和敏感性分析。

输入数据：

- 行业、地区、生产班次、年度需求、采购计划
- 年用电量、月用电量、日负荷曲线、最大需量、峰平谷比例
- 尖峰、高峰、平段、低谷、深谷、基本电费、需量单价

## 9. 验收标准

### 9.1 Skill 验收

- 每个 skill 都有明确输入、处理步骤、输出和错误处理。
- 需求不完整时，不能生成正式方案。
- 储能 ROI 数据不足时，不能输出正式投资回报结论。
- AI 输出的外发材料必须带审核状态。
- 无依据回答必须提示“暂无依据，需专家确认”。
- 每次执行必须能写入日志和关键记忆。

### 9.2 Cron 验收

- 每个 cron 有清晰频率、输入数据源、输出报告和失败重试策略。
- 日报、周报、月报、季报、年报能按固定结构生成。
- FAQ、缺失项、ROI 偏差、模板使用效果能自动沉淀。
- 定时任务不能覆盖人工审核结论，只能生成建议和待确认项。

### 9.3 `memory.md` 验收

- 包含平台定位、核心原则、准入规则、评分规则、ROI 规则、模板规则、权限矩阵、数据字典。
- FAQ 和专家修正能持续追加。
- 复盘结论能反向更新销售打法、模板、标准包和 ROI 默认参数。
- 记忆中必须区分“已确认事实”“默认假设”“待专家确认”“历史复盘结论”。

## 10. 不做或暂不做

本阶段不优先做完整平台 UI。

暂不把 AI 作为最终审核人。

暂不允许 AI 自由生成方案目录。

暂不允许 ROI 结果在数据不足时作为正式投资结论外发。

暂不把竞品对比、价格成本、合同级承诺做成无需审核的自动输出。

## 11. 一句话真实需求

基于 OpenClaw 做一套产品专家 AI 工作流包：用 `skill` 完成需求采集、需求准入、方案生成、ROI 测算、客户赋能、交付运维和经营复盘；用 `openclaw cron` 自动做同步、巡检、FAQ 沉淀、周期报告和模型校准；用 `memory.md` 保存业务规则、数据字典、FAQ、审核口径和复盘经验，让销售、方案、赋能、交付、FAE 和管理层形成可持续优化的闭环。
